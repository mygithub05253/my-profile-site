## 1. 홈 랜딩 (FR-V01)

### Step 1. 기능 개요 (Overview)
- 기능명: 30초 스캔형 홈 랜딩
- User Story: "채용담당자는 랜딩 후 30~60초 내에 기술 스택·핵심 프로젝트·성과를 파악하고 연락처에 도달할 수 있다." (UC-01)
- 대상: 사용자 A·B

### Step 2. 조건 및 상태 (Conditions)
- Pre: content-hub clone에 성공한 프로덕션 빌드 존재
- Post: 스크롤만으로 Hero → Contact 전 섹션 도달, 각 섹션에서 상세로 분기 가능

### Step 3. UI/UX 및 네비게이션
- 섹션 순서(고정): Hero → About 강점 4카드 → Stack(주력/학습) → Featured Projects(3~4) → Records 타임라인 → Blog 최신 3 → Contact
- Hero: h1 `--text-4xl` + accent 태그라인 + muted 1줄 + [프로젝트 보기](Primary) / [연락하기](Secondary)
- 헤더: 홈=섹션 앵커, 서브=라우트 이동. 모바일 햄버거 → 풀스크린 오버레이
- 스크롤 진입 fade-up 8px 1회, stagger 80ms (reduced-motion 시 비활성)

### Step 4. 상세 로직 (Logic)
- Featured = `projects.filter(featured && status=published).sort(order)` 상위 4건
- Blog 최신 3 = 노출 조건(`visibility=public AND status ∈ {published, synced}`) 통과분 date DESC
- Records/Stack/Hero 문구 = `src/data/*` 정적 소스 (관리 경로는 D-1 결정 대기 — 검토보고서)

### Step 5. API/Data
- 빌드 타임 Velite 조회만 — 런타임 API 호출 0건

### Step 6. 예외 처리 (Exception)
- projects 0건 / blog 0건: 해당 섹션 자체 비노출 (빈 껍데기 금지)

### Step 7. Non-functional
- Hero는 이미지 미사용(텍스트+토큰 배경)으로 LCP 확보

## 2. 프로젝트 탐색 — 목록 + 필터 (FR-V02)

### Step 1. Overview
- 기능명: `/projects` 목록 및 6종 필터
- User Story: "방문자는 관심 축(도메인/참여 형태)으로 프로젝트를 걸러 보고 카드에서 상세로 진입한다." (UC-02)

### Step 2. Conditions
- Pre: Velite projects 컬렉션 ≥ 1건 (status=published)
- Post: 필터 상태가 URL에 반영되어 공유·뒤로가기에 보존

### Step 3. UI/UX
- 필터 칩(단일 선택): `All | AI·Data | Finance | Fullstack | Personal | Team` — 활성 = `--accent-soft` 배경 + `--accent` 텍스트
- 그리드: lg 3열 / md 2열 / 모바일 1열. 카드 = 썸네일(16:9) + 칩 + 제목 + summary + 스택 아이콘 3+N
- Private: 우상단 `--warning` "비공개" 배지. 카드 전체 단일 링크(중첩 링크 금지)

### Step 4. Logic
- 매핑: AI·Data/Finance/Fullstack → `category.includes(x)` / Personal/Team → `scope === x` / All → 전체
- 정렬: featured DESC → order ASC
- URL: `?filter=ai-data`. 쿼리 → 상태 초기화(딥링크), 칩 클릭 → `router.replace`
- 필터링은 클라이언트 (11건 규모, SSG 전량 내장)

### Step 5. API/Data
- 없음(SSG). 소비 필드: category, scope, featured, order, summary, stack, thumbnail, repoVisibility

### Step 6. Exception
- 빈 필터 결과: "해당 카테고리 프로젝트 준비 중" (레이아웃 붕괴 금지)
- 유효하지 않은 `?filter=`: All 폴백 (에러 미노출)

### Step 7. Non-functional
- 칩 전환 레이아웃 시프트 최소화, 전환 `--motion-fast`

## 3. 프로젝트 상세 (FR-V03)

### Step 1. Overview
- 기능명: `/projects/[slug]` 정적 상세
- User Story: "방문자는 사이트 안에서 문제 정의→구현→성과→트러블슈팅을 읽고, 필요 시 보조 버튼으로 GitHub/Demo 이동." (UC-02)

### Step 2. Conditions
- Pre: 해당 slug의 published MDX 존재 (`generateStaticParams` 전 slug SSG)
- Post: OG 메타 완비 — 공유 시 썸네일 노출

### Step 3. UI/UX
- breadcrumb(← Projects) → h1+배지 → 칩+기간(mono)+역할 → [GitHub에서 보기][Live Demo](Secondary) → 대표 이미지 → MDX 본문 → 이전/다음 내비
- MDX 컴포넌트(앵커·하이라이트·콜아웃·캡션)는 블로그와 공유. 성과 수치 = `--font-mono` + `--success`

### Step 4. Logic
- `repoVisibility=private` ⇒ GitHub 버튼 미렌더 + "비공개 프로젝트" 배지 (P-3)
- demo 빈 값 ⇒ 버튼 생략
- 이전/다음 = **전역 정렬(featured→order) 기준, 현재 필터와 무관** — 비목표 확정 (검토 F-8). 첫/끝 단방향

### Step 5. API/Data
- OG: `opengraph-image.tsx` (블로그 패턴 재사용, 다크 토큰 고정, css2 `text=` 서브셋)

### Step 6. Exception
- 미존재 slug: 404. draft는 params 생성 제외. 썸네일 없음 → next/og 대체

### Step 7. Non-functional
- 본문 최대폭 72rem, 이미지 lazy

## 4. 블로그 (FR-V04)

- 기존 구현 유지 + PR-B 재스킨만. 노출 조건·permalink(`encodeURIComponent`)·OG 불변
- 예외: clone 실패 = 빌드 실패 = 이전 배포 유지 (아키텍처 F-3 시나리오)

## 5. 테마 토글 (FR-V05)

- 다크 기본. `prefers-color-scheme` 초기값 → 수동 토글 시 `localStorage` 우선
- FOUC 방지: `<head>` 인라인 스크립트가 첫 페인트 전 `data-theme` 결정
- localStorage 불가(시크릿 등): 시스템 값 폴백, 토글은 세션 한정. OG는 테마 무관 다크 고정

## 6. OG 썸네일 (FR-V06)

- 대상: 홈·블로그 글별(운영 중) + **프로젝트별(PR-C 신설)**. thumbnail frontmatter 있으면 우선
- 예외: 폰트 서브셋 실패 → 시스템 폰트 폴백(빌드 지속)
- 운영 노트: SNS 크롤러 캐시로 갱신 지연 가능 (검토 F-10)

## 7. 반응형·접근성 (FR-V07)

- <768 1열·햄버거 / ≥768 2열·인라인 내비 / ≥1024 3열·72rem 중앙
- AA 대비, 포커스 링, 시맨틱 랜드마크 + 스킵 링크, 오버레이 포커스 트랩, `prefers-reduced-motion` 전체 비활성

## 8. SEO (FR-V08)

- 페이지별 `generateMetadata`(title·description·OG·twitter)
- sitemap.xml: 정적 페이지 + blog 49 + **projects 11 포함, 한글 slug 인코딩 일관** (검토 F-9). robots.txt 정적

## 공통 에러·로딩 정책 (사용자 측)

- 런타임 API 없음 → 에러 표면은 **404 + 빈 상태 UI 2종**뿐. 빈 상태 문구 표준은 §1·§2
- 5xx 없음(정적 서빙). 배포 실패는 방문자에게 비가시(이전 배포 유지)
