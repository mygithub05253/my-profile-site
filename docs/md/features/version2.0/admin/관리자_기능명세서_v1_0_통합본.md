## 9. 레포 완성 신호 → velog 초안 자동 생성 (FR-M9, P2) — 2026-07-05 신설

### 배경
- 사용자 아이디어: "GitHub에 레포지토리를 만들면 자동으로 velog에 들어갈 글을 생성해서 올려주는 기능"
- **레포 생성(create) 자체를 트리거로 쓰지 않는다** — 생성 직후에는 내용이 없어 빈 글이 나간다. 대신 "완성 신호"를 트리거로 채택.
- **제약**: 유료 LLM API 사용 불가(학생, 무비용) → 요약은 GitHub REST API 메타데이터 + **템플릿 문장 치환**으로 생성. 관리자가 다듬어 승인해야 실제 발행되므로 품질은 사람이 최종 보장.

### Step 1. Overview
- User Story: "관리자가 개인 레포에 완성 신호(예: 특정 파일 push 또는 topic 추가)를 남기면, 그 레포 소개 글감이 자동으로 content-hub `drafts/`에 초안으로 올라온다. 관리자가 다듬어 승인하면 기존 파이프라인(A11)이 velog에 발행한다."

### Step 2. Conditions
- Pre: 대상 레포가 profile-admin에 **사전 등록**(FR-M10의 등록 레포 목록 재사용) + 해당 레포에 완성 신호 존재
- Post: content-hub `drafts/{repo-slug}-intro.md` 신규 커밋(PR) 생성, status: draft, source: github-repo

### Step 3. 트리거 설계
- 완성 신호 후보 (택1, P2 구현 시 확정): ① 레포 topics에 `portfolio-ready` 라벨 추가 ② README에 `<!-- READY-FOR-VELOG -->` 마커 존재 + push
- 등록 레포 각각에 **GitHub Actions 워크플로 1개**(`.github/workflows/notify-content-hub.yml`, 공용 템플릿을 레포마다 붙여넣기 — 무료, GitHub Actions 무료 티어 한도 내)를 두고, push 시 신호 조건을 체크해 content-hub로 `repository_dispatch(event_type: repo-ready)` 전송 (payload: repo full_name)

### Step 4. Logic — 초안 생성 (무료, LLM 미사용)
- content-hub 측 워크플로가 dispatch 수신 → GitHub REST API로 대상 레포의 `description`·`language`·`topics`·최근 커밋 5건 메시지·`README.md` 첫 문단(정규식으로 첫 heading 다음 문단 추출) 조회
- 아래 **고정 템플릿**에 값만 치환해 마크다운 생성 (문장 생성 LLM 없음):
  ```
  ---
  title: "{repo_name} 개발 회고"
  slug: "{repo-slug}-intro"
  type: retrospective
  source: velog
  status: draft
  tags: [{language}, {topics[0..2]}]
  ---
  # {repo_name}

  {description}

  ## 사용 기술
  {language}, {topics 목록}

  ## 최근 작업
  {최근 커밋 메시지 5건 불릿}

  ## 프로젝트 소개
  {README 첫 문단}

  > 이 초안은 자동 생성되었습니다. 발행 전 내용을 다듬어주세요.
  ```
- 생성 파일을 `content-hub` `drafts/{repo-slug}-intro.md`로 커밋 → PR 생성(자동 병합 안 함 — 사람이 열람 후 병합해야 드래프트로 인입, 무의미한 초안 유입 방지)

### Step 5. API/Data
- content-hub 신규 워크플로: `.github/workflows/generate-draft.yml` (트리거: `repository_dispatch: [repo-ready]`)
- 각 등록 레포에 배포할 공용 액션: 별도 레포(`mygithub05253/portfolio-signal-action` 또는 mygithub05253 프로필 레포 내 공용 액션)로 재사용 가능하게 구성 — 레포마다 워크플로 파일 1줄 복붙 수준으로 최소화
- 관리자는 profile-admin의 FR-M19(블로그 초안 관리) 화면에서 이 초안을 그대로 편집·ready 승격 (신규 UI 불요 — 기존 화면 재사용)

### Step 6. Exception
- dispatch 실패(레포 미등록·토큰 권한 부족): 워크플로 실패 로그만 남기고 종료(사이트 영향 0)
- README 파싱 실패(첫 문단 추출 안 됨): 해당 섹션 생략, `description`만으로 초안 생성 — 빈 글보다 부실한 초안이 낫다는 원칙(관리자가 항상 다듬음)

### Step 7. Non-functional
- **비용 0**: GitHub Actions 무료 티어(개인 Public 레포 무제한 분) + GitHub REST API(토큰 인증, 무료) + 템플릿 치환(순수 코드). LLM 호출 없음
- 관계: FR-M19(드래프트 관리)의 상류 공급 단계. FR-M19가 최종 승인·발행을 담당하므로 이 기능은 **초안 생성만** 책임진다(경계 분리)

## 10. 레포 지정 → 프로젝트 CRUD 자동 스캐폴딩 (FR-M10, P1) — 2026-07-05 신설

### 배경
- 사용자 아이디어: "관리자 기능에서 레포지토리를 지정하거나 프로젝트를 지정해줄 경우 자동으로 CRUD 양식에 맞게 생성해주는 기능"
- **제약**: 유료 LLM 없음 → 필드 자동 채움은 GitHub REST API 메타데이터 + 규칙 기반 매핑. 자동 추론이 어려운 필드(category·scope·featured 등 취향 판단)는 관리자가 폼에서 직접 선택하도록 설계(억지 자동화 금지)

### Step 1. Overview
- User Story: "관리자는 GitHub 레포 하나를 지정하면, 그 레포의 메타데이터로 프로젝트 등록 폼이 자동으로 채워진 상태에서 시작해, 빈 칸만 다듬어 저장한다."

### Step 2. Conditions
- Pre: 인증 완료(FR-M16), 대상 레포가 관리자 GitHub 계정 소유(또는 등록된 협업 레포)
- Post: FR-M17 편집 폼이 자동 채움 상태로 열림 — 저장 전까지 content-hub 커밋 없음(FR-M17 저장 흐름 그대로 재사용, 이 기능은 프리필 단계만 추가)

### Step 3. UI/UX
- FR-M17 목록 화면에 [새 프로젝트] 버튼 옆 **[GitHub에서 가져오기]** 버튼 추가
- 클릭 시 관리자 GitHub 레포 드롭다운(REST API `GET /user/repos` 조회 결과) 또는 `owner/repo` 직접 입력
- 선택 즉시 FR-M17 편집 폼으로 이동, 아래 자동 매핑 값이 프리필된 상태로 표시(모든 필드 수정 가능)

### Step 4. Logic — 자동 매핑 규칙 (LLM 없음, 규칙 기반)
| projects 필드 | 소스 | 매핑 규칙 |
|---|---|---|
| title | `repo.name` | 그대로 제안 (수정 가능) |
| slug | `repo.name` | kebab-case 변환 제안 |
| github | `repo.html_url` | 그대로 |
| repoVisibility | `repo.private` | true→private, false→public |
| period | `repo.created_at`~`repo.pushed_at` | `YYYY.MM ~ YYYY.MM` 포맷 변환 (수정 가능) |
| stack | `repo.language` + `repo.topics` | 사전 정의된 언어→스택 매핑 테이블(예: `TypeScript`→`Next.js` 후보 제안, 확정 아님) + topics 그대로 태그 제안 |
| summary | `repo.description` 우선, 없으면 README 첫 문단 | 그대로 제안(관리자 다듬기 전제, 문장 생성 안 함) |
| category / scope / featured / order | — | **자동 추론 안 함** — 관리자가 직접 선택(취향·판단 영역). scope만 `repo.collaborators` API 조회 수로 personal(1)/team(2+) 후보를 참고 배지로만 표시 |
| thumbnail | — | 미채움 — 기존 이미지 업로드(FR-M22)로 관리자가 별도 첨부 |

### Step 5. API/Data
- 신규 서버 라우트: `GET /api/admin/github-import?repo={owner}/{repo}` — GitHub REST API `GET /repos/{owner}/{repo}`, `GET /repos/{owner}/{repo}/readme`, `GET /repos/{owner}/{repo}/languages` 조합 호출 후 위 매핑 규칙 적용해 projects frontmatter 초안 JSON 반환
- README 첫 문단 추출: 마크다운 파서(예: `remark`, 오픈소스 무료)로 AST 파싱 후 첫 heading 다음 paragraph 노드 추출 — 정규식보다 안정적
- 이 라우트는 **읽기 전용**(GitHub 조회만, content-hub 쓰기 없음) — 저장은 반드시 FR-M17의 기존 PR+auto-merge 흐름을 통과해야 함(스캐폴딩이 검증을 우회하지 않음)

### Step 6. Exception
- 레포 404/권한 없음: "레포를 찾을 수 없습니다" 안내, 수동 입력으로 폴백(빈 폼)
- README 없음/파싱 실패: summary 필드만 `repo.description`으로 채우고 나머지는 공백 — 자동 스캐폴딩 실패가 수동 등록을 막지 않음(항상 폴백 가능)
- rate limit(GitHub REST API 5,000/h, 인증 시): 1인 관리자 사용 패턴에서 도달 가능성 낮음 — 도달 시 표준 에러 배너 + 재시도 안내

### Step 7. Non-functional
- **비용 0**: GitHub REST API(무료, 인증 토큰) + `remark`(오픈소스 npm) + 규칙 기반 매핑(순수 코드). LLM 호출 없음
- 관계: FR-M17(프로젝트 CRUD)의 **입력 보조 단계**. 저장 검증(M18)·충돌 처리(M21)·이미지(M22)는 기존 FR-M17 경로를 그대로 통과하므로 신규 검증 로직 불요(경계 분리)

## 16. 관리자 인증 (FR-M16)

### Step 1. Overview
- User Story: "관리자(본인)는 GitHub 계정으로 로그인해 관리 화면에 진입한다. 타인은 로그인해도 진입 불가."

### Step 2. Conditions
- Pre: GitHub OAuth App 등록(profile-admin 전용), allowlist에 본인 GitHub id 1건
- Post: 세션 확립 → 전 관리 라우트 접근 가능

### Step 3~4. UI/UX · Logic
- 로그인 화면 = GitHub 버튼 1개. 로그인 후 프로젝트 목록(A-02)으로
- 미들웨어: 세션 없음 → /login / allowlist 외 → **403 고정 페이지** (사유 미노출)

### Step 5~7. API · Exception · NFR (R-2 반영 확정)
- **Auth.js v5** GitHub Provider, JWT 세션
- allowlist 기준: GitHub **숫자 ID**(`profile.id`) — 사용자명(login)은 변경 가능하므로 사용 금지. env `ALLOWED_GITHUB_ID`
- 3중 방어: ① `signIn` 콜백 — ID 불일치 시 `false` 반환(AccessDenied) ② `jwt`/`session` 콜백에 githubId 저장 ③ 미들웨어 `authorized` 콜백 — 세션 존재 + githubId == allowlist 재확인
- 403 라우팅: `pages.error: "/auth/error"` → error=AccessDenied 시 403 고정 문구 (사유 미노출)
- JWT 세션 정책: 1인 관리자 특성상 `session.maxAge` 단축(예: 7일). 개별 세션 무효화 불가 한계 인지 — 유출 의심 시 `AUTH_SECRET` 회전으로 전체 무효화
- OAuth 실패·취소: /login 에러 배너. rate limit 불요(1인)

## 17~18. 프로젝트 CRUD + 즉시 검증 (FR-M17·M18)

### Step 1. Overview
- User Story: "관리자는 코드 수정 없이 웹 폼으로 프로젝트를 등록·수정·삭제하고, 저장 즉시 사이트에 자동 반영된다." (UC-04)

### Step 2. Conditions
- Pre: 인증 완료, content-hub `projects/` 존재 (PR-A 병합)
- Post: content-hub 커밋 생성 → 기존 파이프라인이 재빌드 (admin 측 추가 배선 0)

### Step 3. UI/UX
- 목록: 테이블(제목·category·scope·status·featured·수정일) + [새 프로젝트]
- 편집: frontmatter 폼(enum=select, 배열=태그 입력) + MDX 에디터(작성/프리뷰 탭)
- 저장 상태줄: idle → validating → committing → done/error

### Step 4. Logic
- 검증(M18): zod = 스키마 §4 전 규칙(P-1 role 조건부, P-2 category, **P-5 status 2단계**) — 필드 인라인 에러
- slug: 생성 시 title 제안 + 수동 수정, **저장 후 불변** (수정 화면 read-only)
- 커밋 메시지 규약: `admin: {create|update|delete} project {slug}`

### Step 5. API/Data — 저장 방식 **확정: (b) PR + auto-merge** (D-2, 2026-07-04)
- 라우트: 관리자 API 명세서 §6.2 (5개)
- 저장 흐름: `admin/{slug}-{ts}` 브랜치 → **Git Database API 원자 커밋**(blob→tree→commit→ref, 본문+이미지 단일 커밋 — R-1) → PR 생성 → **auto-merge 활성화**(GraphQL `enablePullRequestAutoMerge`, SQUASH) → CI(L1) 통과 시 자동 병합
- 사전 요건(1회 설정): ① 레포 `allow_auto_merge: true` ② main 브랜치 보호에 L1 검증을 required status check로 등록 (요건 0개면 auto-merge 미적용 — R-1) ③ 토큰 권한 `contents: write` + `pull-requests: write`
- 단일 텍스트 파일만 변경 시에도 동일 흐름 (경로 단일화 — Contents API PUT 반복 호출은 원자성 없음)

### Step 6. Exception
- 409(sha 충돌): 재조회 → diff 표시 → [덮어쓰기/취소]
- 400(스키마): 필드 에러 매핑 / 502(GitHub 장애): 작성 내용 localStorage 임시 보존 + 재시도 안내
- (b)안 특유: CI 실패 시 PR 미병합 → admin 대시보드에 "검증 실패" 상태 표시 + PR 링크

### Step 7. Non-functional
- dirty 이탈 경고. 삭제 = 확인 모달 + **slug 재입력** (파괴 조작 2중 확인)

## 21. 충돌 처리 (FR-M21)

- 낙관적 잠금 정본(R-1 보강): 조회 시 sha 보관 → 저장 시 첨부 → 불일치 409 → **최신 sha 재조회 후 지수 백오프(200/400/800ms) 최대 3회 자동 재시도** → 실패 시 diff 수동 해소. sha는 항상 직전 GET에서 취득(캐시 재사용 금지)
- Git Database API 경로에서는 `PATCH /git/refs` 422가 동일 신호 — step 0(최신 SHA 재조회)부터 재시도
- 경합 상대는 타 기기/CI 커밋(1인 운영) — diff 표시로 충분

## 22. 이미지 업로드 (FR-M22, P1) — R-3 반영 확정

- **채택안: content-hub `assets/` git 커밋 유지** — 수백 장 이하·저트래픽 포트폴리오 규모는 R-3 권장 조건("이미지 변경 거의 없음, 트래픽 낮음")에 부합. 외부 서비스(Cloudflare R2/Cloudinary)는 한계 도달 시 전환 (아래 마이그레이션 조건)
- 업로드 파이프라인: 드롭존 → **클라이언트 측 리사이즈(최대 폭 1600px)·WebP 변환·압축** → `assets/{slug}/{filename}.webp` → 본문과 **단일 원자 커밋** (원본 무압축 커밋 금지 — git 히스토리 비대화 방지, R-3)
- 서빙: 사이트 빌드 시 clone된 로컬 파일로 번들 — **raw.githubusercontent.com 핫링크 금지** (CDN 아님·rate limit·캐시 미제어, R-3)
- 크기 가드: 장당 5MB 초과 시 업로드 거부 (Contents API 공식 100MB·실용 50MB 대비 보수 운영)
- 마이그레이션 조건: 레포 1GB 근접 또는 이미지 수정 빈발 시 R2/Cloudinary 전환 — `assets/` 경로 규칙 유지로 URL 일괄 치환 가능하게 설계
