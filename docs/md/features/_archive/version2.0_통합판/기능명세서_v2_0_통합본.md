# 도메인 A. 프로필 사이트 방문자 플로우

## 1. 홈 랜딩 (FR-V01)

### Step 1. 기능 개요 (Overview)
- 기능명: 30초 스캔형 홈 랜딩
- User Story: "채용담당자는 랜딩 후 30~60초 내에 기술 스택·핵심 프로젝트·성과를 파악하고 연락처에 도달할 수 있다." (UC-01)
- 대상: 사용자 A(채용담당자)·B(동료 개발자)

### Step 2. 조건 및 상태 (Conditions)
- Pre: content-hub clone 성공한 프로덕션 빌드 존재
- Post: 스크롤만으로 Hero → Contact 전 섹션 도달. 각 섹션에서 상세(프로젝트·블로그)로 분기 가능

### Step 3. UI/UX 및 네비게이션
- 섹션 순서(고정): Hero → About 강점 4카드 → Stack(주력/학습) → Featured Projects(3~4) → Records 타임라인 → Blog 최신 3 → Contact
- Hero: h1 `--text-4xl` 이름 + accent 태그라인 + muted 1줄 + [프로젝트 보기](Primary)/[연락하기](Secondary)
- 헤더 앵커 내비(홈) / 라우트 내비(서브). 모바일 햄버거 → 풀스크린 오버레이
- 스크롤 진입 fade-up 8px 1회, stagger 80ms (reduced-motion 시 비활성)

### Step 4. 상세 로직 (Logic)
- Featured = `projects.filter(featured).sort(order)` 상위 4건
- Blog 최신 3 = 노출 조건(`public + published|synced`) 통과분 date DESC
- Records/Stack/Hero 문구는 `src/data/*` 정적 소스 (스키마 명세서 §8)

### Step 5. API/Data
- 빌드 타임 Velite 조회만 — 런타임 API 호출 0건

### Step 6. 예외 처리 (Exception)
- projects 0건(초기): Featured 섹션 자체 비노출 (빈 껍데기 금지)
- 블로그 0건: 동일 정책

### Step 7. Non-functional
- NFR-01: Hero는 이미지 미사용(텍스트+토큰 배경)으로 LCP 확보

## 2. 프로젝트 탐색 — 목록 + 필터 (FR-V02)

### Step 1. Overview
- 기능명: `/projects` 목록 및 6종 필터
- User Story: "방문자는 관심 축(도메인/참여 형태)으로 프로젝트를 걸러 보고 카드에서 상세로 진입한다." (UC-02)

### Step 2. Conditions
- Pre: Velite projects 컬렉션 ≥ 1건 (status=published)
- Post: 필터 상태가 URL에 반영되어 공유·뒤로가기에 보존

### Step 3. UI/UX
- 필터 칩(단일 선택): `All | AI·Data | Finance | Fullstack | Personal | Team` — 활성 칩 `--accent-soft` 배경 + `--accent` 텍스트
- 카드 그리드: lg 3열 / md 2열 / 모바일 1열. 카드 = 썸네일(16:9) + 칩 + 제목 + summary + 스택 아이콘 3+N
- Private 프로젝트: 우상단 `--warning` "비공개" 배지

### Step 4. Logic
- 매핑: AI·Data/Finance/Fullstack → `category.includes(x)` / Personal/Team → `scope === x` / All → 전체
- 정렬: featured DESC → order ASC
- URL 동기화: `?filter=ai-data` 형식. 쿼리 → 상태 초기화(딥링크), 칩 클릭 → `router.replace`(히스토리 오염 방지, 단 뒤로가기 복원은 보장)
- 구현: 목록 데이터는 SSG로 전량 내장, 필터링은 클라이언트 (11건 규모 — 재요청 불필요)

### Step 5. API/Data
- 없음 (SSG 내장). frontmatter 소비 필드: category, scope, featured, order, summary, stack, thumbnail, repoVisibility

### Step 6. Exception
- 빈 필터 결과: "해당 카테고리 프로젝트 준비 중" 안내 문구 (레이아웃 붕괴 금지)
- 유효하지 않은 `?filter=` 값: All로 폴백 (에러 미노출)

### Step 7. Non-functional
- 칩 전환 시 레이아웃 시프트 최소화 (그리드 높이 유지 전략), 애니메이션 `--motion-fast`

## 3. 프로젝트 상세 (FR-V03)

### Step 1. Overview
- 기능명: `/projects/[slug]` 정적 상세 페이지
- User Story: "방문자는 사이트 안에서 프로젝트의 문제 정의→구현→성과→트러블슈팅을 읽고, 필요 시 보조 버튼으로 GitHub/Demo로 이동한다."

### Step 2. Conditions
- Pre: 해당 slug의 published MDX 존재 (`generateStaticParams`로 전 slug SSG)
- Post: OG 메타 완비 → 공유 시 썸네일 노출

### Step 3. UI/UX
- 구조: breadcrumb(← Projects) → h1+배지 → 칩(category·scope)+기간(mono)+역할 → [GitHub에서 보기][Live Demo](Secondary) → 대표 이미지 → MDX 본문 → 이전/다음 프로젝트 내비
- MDX 컴포넌트: 제목 앵커·코드 하이라이트·콜아웃·이미지 캡션 — **블로그와 공유** (MDXComponents 단일 소스)
- 성과 수치: `--font-mono` + `--success` 강조

### Step 4. Logic
- `repoVisibility === "private"` ⇒ GitHub 버튼 렌더링 자체 생략 + "비공개 프로젝트" 배지 (P-3 제약)
- demo 빈 값 ⇒ Demo 버튼 생략
- 이전/다음 = 목록 정렬 기준(featured→order) 순환 없음(첫/끝은 단방향)

### Step 5. API/Data
- OG: `opengraph-image.tsx` — 기존 블로그 패턴 재사용, 다크 토큰 고정, 한글 폰트는 css2 `text=` 서브셋

### Step 6. Exception
- 미존재 slug: 404 (notFound()). draft 상태 프로젝트는 params 생성 제외
- 썸네일 없음: next/og 생성 이미지로 대체

### Step 7. Non-functional
- 본문 최대폭 `72rem` 내 콘텐츠 컬럼, 이미지 lazy

## 4. 블로그 (FR-V04)

- 기존 구현 유지 + PR-B에서 신규 토큰 재스킨만 수행. 노출 조건·permalink(`encodeURIComponent`)·OG 불변
- 예외: content-hub clone 실패 시 빌드 실패 = 이전 배포 유지 (F-3)

## 5. 테마 토글 (FR-V05)

### Step 1~3 (Overview·Conditions·UI/UX)
- 다크 기본. `prefers-color-scheme` 초기값 → 수동 토글 시 `localStorage` 우선
- 헤더 우측 sun/moon 아이콘, 전환 `--motion-base 250ms`

### Step 4~6 (Logic·Data·Exception)
- FOUC 방지: `<head>` 인라인 스크립트가 첫 페인트 전 `data-theme` 결정

## 6. OG 썸네일 (FR-V06)

# 도메인 C. 관리자 웹 profile-admin (FR-M16~21)

## 16. 관리자 인증 (FR-M16)

### Step 1. Overview
- User Story: "관리자(본인)는 GitHub 계정으로 로그인해 콘텐츠 관리 화면에 진입한다. 타인은 로그인해도 진입 불가."

### Step 2. Conditions
- Pre: GitHub OAuth App 등록, allowlist에 본인 id 1건
- Post: 세션 확립 → 전 관리 라우트 접근 가능

### Step 3~4. UI/UX·Logic
- 로그인 화면 = GitHub 버튼 1개 (미니멀). 로그인 후 프로젝트 목록으로
- 미들웨어: 세션 없음 → /login, 세션 있으나 allowlist 외 → 403 고정 페이지(정보 최소 노출)

### Step 5~7. API·Exception·NFR
- NextAuth GitHub Provider. 세션 전략 JWT. PAT/토큰은 서버 env만 (NFR-03)
- OAuth 실패·취소: /login 에러 배너. rate limit 고려 불필요 (1인)

## 17~18. 프로젝트 CRUD + 즉시 검증 (FR-M17·M18)

### Step 1. Overview
- User Story: "관리자는 코드 수정 없이 웹 폼으로 프로젝트를 등록·수정·삭제하고, 저장 즉시 사이트에 자동 반영(재빌드)된다." (UC-04)

### Step 2. Conditions
- Pre: 인증 완료. content-hub `projects/` 존재 (PR-A 병합)
- Post: content-hub 커밋 생성 → 기존 파이프라인이 재빌드 (admin 측 추가 배선 0)

### Step 3. UI/UX
- 목록: 테이블(제목·category·scope·status·featured·수정일) + [새 프로젝트]
- 편집: frontmatter 폼(스키마 §4 필드별 위젯 — enum은 select, 배열은 태그 입력) + MDX 본문 에디터(프리뷰 탭)
- 저장 버튼 상태: idle → validating → committing → done/error

### Step 4. Logic
- 검증(M18): zod 스키마 = 스키마 명세서 §4 (P-1 role 조건부 필수, P-2 category 등) — 필드 단위 인라인 에러
- slug: 생성 시 title에서 제안 + 수동 수정 가능, **저장 후 불변** (수정 화면에서 read-only)
- 커밋 메시지 규약: `admin: {create|update|delete} project {slug}`

### Step 5. API/Data
- 연동 API 명세서 §6.2의 5개 라우트. 저장 = GitHub Contents PUT (base64+sha)

### Step 6. Exception
- 409(sha 충돌): 자동 재조회 → 변경분 diff 표시 → 사용자 선택(덮어쓰기/취소)
- 400(스키마): 필드 에러 매핑. 502(GitHub 장애): 작성 내용 localStorage 임시 보존 후 재시도 안내

### Step 7. Non-functional
- 편집 중 이탈 경고(dirty check). 삭제는 확인 모달 + slug 재입력 (파괴 조작 2중 확인)

## 19. 블로그 초안 관리 (FR-M19, P1)

- drafts/ 목록·편집(FR-M17 폼 재사용, posts 스키마) · [발행 준비] = drafts→posts 이동 커밋(=ready 승격, A11 자동 발행 트리거)
- 승격 확인 모달에 "velog에 자동 발행됩니다" 명시 (의도치 않은 발행 방지)

## 20. 발행 대시보드 (FR-M20, P2)

- GitHub Actions API로 velog-publish·deploy-site·blog-post 최근 실행 표시 (성공/실패·시각·로그 링크)
- 글별 상태 매트릭스: status × velog_url 유무 × 사이트 노출 여부

## 21. 충돌 처리 (FR-M21)

- 낙관적 잠금 정책 정본: 조회 시 sha 보관 → 저장 시 첨부 → 불일치 409 → 1회 자동 재시도(재조회) → 실패 시 수동 해소 UI
- 동시 편집자는 본인뿐이므로(1인) 충돌은 주로 "다른 기기/CI 커밋"과의 경합 — diff 표시로 충분

# 공통 에러·로딩 정책

1. **사이트(정적)**: 런타임 API 없음 → 에러 표면은 404/빈 상태 UI 2종뿐. 빈 상태는 안내 문구 표준(2·6장)
2. **admin**: 모든 fetch에 로딩 스피너(200ms 지연 노출)·에러 배너·재시도 버튼 3종 세트. 4xx는 원인 문구, 5xx/502는 일반 문구+재시도
3. **파이프라인**: 실패는 GitHub Actions 알림으로 수신. 자동 재시도는 네트워크성 1회만 (API 명세서 §8)
