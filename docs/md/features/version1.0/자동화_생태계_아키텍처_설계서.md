## 1. 목표 및 요구사항 정의

### 1.1 비전
"한 번 쓰면 모든 곳에 반영된다" — Obsidian/Notion에서 글을 쓰면 velog 발행, GitHub 백업,
프로필 사이트 블로그 섹션, GitHub 프로필 README까지 자동으로 동기화되는 개인 브랜딩 생태계.

### 1.2 기능 요구사항 (FR)

| ID | 요구사항 | 우선순위 |
|----|----------|----------|
| FR-01 | Obsidian Vault 또는 Notion DB에서 작성한 글을 정규화된 Markdown으로 GitHub에 수집 | P0 |
| FR-02 | 수집된 글을 velog에 자동(또는 반자동) 발행 | P0 |
| FR-03 | velog에 직접 쓴 글을 역방향 백업 (기존 velog-backup 개선) | P0 |
| FR-04 | 썸네일 파이프라인: 프롬프트 자동 생성 → 이미지 투입 시 자동 반영 | P1 |
| FR-05 | 프로필 사이트(Next.js)에 블로그/회고/프로젝트 섹션 자동 반영 | P0 |
| FR-06 | GitHub 프로필 README에 최신 글 목록 자동 갱신 | P1 |
| FR-07 | 회고/프로젝트 소개 등 lova-clover식 콘텐츠 유형 지원 (frontmatter의 type 필드) | P1 |
| FR-08 | 발행 이력/실패 로그 추적 (Actions Summary + 커밋 로그) | P2 |

### 1.3 비기능 요구사항 (NFR)

- **NFR-01 서버리스**: 상시 서버 없이 GitHub Actions만으로 동작 (비용 0원 목표)
- **NFR-02 단일 진실 공급원(SSOT)**: 콘텐츠 원본은 반드시 한 곳(콘텐츠 레포)에만 존재
- **NFR-03 장애 격리**: velog API(비공식) 장애가 사이트 배포를 막지 않아야 함
- **NFR-04 보안**: 모든 토큰은 GitHub Secrets, PAT은 fine-grained + 최소 권한
- **NFR-05 무료 플랜 준수**: 개인 계정 다중 레포 (Organization 미사용)

### 1.4 핵심 리스크

| 리스크 | 영향 | 대응 |
|--------|------|------|
| velog GraphQL API는 **비공식** — 토큰 만료/스펙 변경 가능 | 자동 발행 중단 | 반자동 모드 폴백 (velog_ready/ 폴더에 완성본 생성 → 수동 붙여넣기), 실패 시 Actions 알림 |
| Notion API 블록 파싱 복잡도 (중첩 블록, 이미지 만료 URL) | 변환 오류 | oneokiwa 사례의 재귀 파싱 + 이미지 로컬 다운로드 방식 채택 |
| Obsidian ↔ Notion 이중 작성 시 충돌 | 콘텐츠 중복 | frontmatter `source` 필드 + slug 기준 중복 감지 |

## 3. 데이터 흐름 (End-to-End)

### 3.1 순방향: 작성 → 발행

```
[Obsidian Vault]──(obsidian-git 플러그인, 자동 commit/push)──┐
                                                            ├→ content-hub
[Notion DB (Status=Ready)]──(notion-sync.yml, cron 30분)────┘      │
                                                                   ▼
                                              ① 정규화: frontmatter 검증/보강 (slug, date, tags, type)
                                              ② 썸네일 검사:
                                                  없음 → thumbnails/prompts/{slug}.md 프롬프트 생성
                                                        + 발행 보류(draft) 또는 기본 썸네일로 발행 (설정)
                                                  있음 → frontmatter thumbnail 필드 주입
                                              ③ velog-publish.yml → velog GraphQL API 발행
                                                  실패 시 → velog_ready/에 완성본 + Actions 경고
                                              ④ dispatch-downstream.yml
                                                  → my-profile-site 재빌드 (블로그 섹션 갱신)
                                                  → mygithub05253 README 갱신
                                              ⑤ Notion DB에 결과 기입 (Published, velog URL)
```

### 3.2 역방향: velog 직접 작성분 회수

```
velog에 직접 작성 → velog-backup (주간 cron, RSS 파싱)
                 → velog-posts/ 커밋
                 → content-hub에 없는 slug 감지 시 content-hub로 자동 PR
                 → 병합 시 SSOT 일원화 완성
```

### 3.3 썸네일 반자동 파이프라인 (사용자 개입 1회)

1. 새 글 감지 시 Actions가 글 요약 기반 **GPT Image 2.0 프롬프트**를 `thumbnails/prompts/{slug}.md`로 생성
2. 사용자가 GPT/Codex에 프롬프트 입력 → 생성된 이미지를 `thumbnails/{slug}.png`로 커밋
3. `thumbnail-attach.yml`이 push 감지 → frontmatter 갱신 → velog 글 수정 API 호출 + 사이트 재빌드

> 향후 Claude API(또는 이미지 생성 API) Secrets 등록 시 2번 단계도 완전 자동화 가능 — Phase 4 과제.

## 10. Phase 0 리서치 반영 — 기술 스펙 확정 (2026-07-03, 설계 확정)

리서치 1~4 결과를 반영한 최종 확정 사항. **결론: 완전 자동 발행은 조건부 가능 → 결정 2(완전 자동 + 폴백) 유지 확정.**

### 10.1 velog 발행 모듈 확정 스펙 (리서치 1)

- 엔드포인트: 쓰기 `https://v3.velog.io/graphql` (writePost/editPost), 이미지 `https://v3.velog.io/api/files/v3/upload`
- 인증: **Cookie 방식만** (`Cookie: access_token=...; refresh_token=...`). Bearer 헤더 금지
- 토큰 전략: GitHub Secret `VELOG_REFRESH_TOKEN`(TTL 약 30일)만 보관 → 매 실행 시 `currentUser` 쿼리로 access_token 자동 재발급 (Set-Cookie 파싱)
  - **운영 규칙: refresh_token 30일 주기 수동 갱신** — 만료 2~3일 전 Actions가 경고 이슈 자동 생성
- 필수 준수 사항 (silent failure 방지):
  - `tags: []` 생략 금지 (required 배열), `meta: {}` 필수, `operationName` 필드 필수
  - `writePost`가 `null` 반환 시 **성공 아님** — 반드시 실패 처리 + velog_ready/ 폴백 + Actions 경고
- **속도 제한: 5분 내 공개 글 10개 초과 시 서버가 자동 비공개 처리** → 공백기 백필(소급 발행) 시 글 사이 60초 이상 간격 스로틀 필수
- 참고 구현체: `seongwon030/velog_mcp`(토큰 갱신 + 일일 health check), `stoneHee99/velog-mcp` v0.2.2+

### 10.2 콘텐츠 파이프라인 확정 스펙 (리서치 2)

- **Notion 이미지는 수집 즉시 다운로드** (presigned URL 1시간 만료) → `assets/{slug}/` 저장 + 경로 치환. Notion rate limit 3 req/s 준수
- **status 3단계**: `draft → ready → synced`. **git push 성공 후에만 `synced` 전환** (실패 시 다음 실행에서 자동 재처리)
- **slug는 불변(immutable)**: 발행 후 변경 금지, 변경 필요 시 frontmatter `aliases`로 리다이렉트
- **자동화 커밋은 `auto-sync` 브랜치 전용** (ironpark 패턴): 수동 작업(main)과 충돌 원천 차단. 빌드 시 병합
- **동기화 레이어 단일 원칙 재확인**: Vault는 Google Drive만, git은 content-hub만 (obsidian-git 미사용 확정 — 9.2 결정 타당성 검증됨)
- 파이프라인이 처리한 글 수를 로깅하고 **0건 시 알림** (Notion 속성명 변경 등 silent failure 감지)

### 10.3 프로필 사이트 확정 스펙 (리서치 3)

- 콘텐츠 계층: **Velite** 채택 (`contentlayer` 원본·`next-mdx-remote` 원본은 방치/아카이브로 금지)
  - Turbopack 비호환 → 빌드 스크립트 `velite build && next build` 순서 고정
- velog 최신 글: 서버 컴포넌트 + `fetch(RSS, { next: { revalidate: 3600 } })` ISR
