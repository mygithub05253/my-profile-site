## 배경

세션 14에서 프로젝트 MDX 10건(draft)을 작성하며 확인해보니 게재 대상 레포 대부분(EST-CAMP-AI-Quant, Emoji-Diary-Final, ember 등)이 README에 커버 이미지·다이어그램을 갖고 있었음. FR-M10(github-import)은 현재 README 첫 문단 텍스트만 추출하고 이미지는 다루지 않으며, FR-M22(이미지 업로드)는 로컬 파일 업로드만 지원함. 사용자가 "README 이미지를 프리필 후보로 활용하면 10건 draft 검수 결과가 더 좋아질 것"이라 판단해 검수/전환 작업(①)보다 이 기능(③) 설계·구현을 먼저 진행하기로 결정.

## 사용자 결정 3건 (2026-07-06)

1. **미리보기**: GitHub raw URL로 브라우저에 바로 `<img>` 미리보기(저장 안 됨, 일시적) — FR-M22의 "raw 핫링크 금지"는 실제 발행 콘텐츠 얘기라 별개로 취급
2. **후보 범위**: 개수 제한·도메인 필터링 없이 README에서 발견되는 이미지 전부 후보로 노출. UI는 쇼핑몰 상세 이미지 패턴(가로 스크롤 썸네일 스트립 → 클릭 시 확대 모달 + 캡션 + 이전/다음 넘기기)으로 다수 이미지를 감당
3. **채택 동작·적용 범위**: "채택"하면 기존 이미지 업로드 목록(StagedImage[])에 합류만 함(thumbnail 필드 자동 입력 없음, 관리자가 "경로 복사"로 직접 지정). 프로젝트 생성 화면과 기존 프로젝트 편집 화면 양쪽 모두에 배치(10건 기존 draft 개선이 목적이므로 편집 화면 필수)

## 아키텍처

### 서버

- `src/lib/github-import.ts`에 `listReadmeImageCandidates(owner, repo): Promise<{url: string; alt: string}[]>` 추가
  - `octokit.repos.get`으로 `default_branch` 조회 + `octokit.repos.getReadme`로 README 원문 확보
  - remark AST 전체 트리 순회(첫 문단만 보던 기존 `extractFirstParagraphAfterHeading`와 달리 전체 탐색 필요) — mdast `image` 노드(마크다운 `![]()` 문법) + `html` 노드(README에 흔한 `<img src="...">` 뱃지/배너 블록, 정규식으로 `src`/`alt` 속성만 targeted 추출 — HTML 파서 신규 의존성 추가 안 함) 양쪽 다 수집
  - 상대경로(`docs/images/x.png`, `./assets/y.png` 등)는 `https://raw.githubusercontent.com/{owner}/{repo}/{default_branch}/{path}`로 절대화, 이미 `http(s)://`인 경우 그대로 사용
  - URL 기준 dedupe. 필터링·cap 없음(사용자 결정 2)
- 신규 API `GET /api/admin/github-import/images?repo={owner}/{repo}` — 읽기 전용, 위 함수 호출 결과 그대로 반환
- 신규 API `GET /api/admin/github-import/fetch-image?url={encoded}` — 서버 프록시
  - **필요 이유**: 채택된 이미지를 기존 압축 파이프라인(리사이즈 1600px·WebP 변환·5MB 가드)에 태우려면 `<canvas>`로 픽셀을 읽어야 하는데, CORS 헤더가 없는 외부 호스트(뱃지 사이트 등) 이미지를 그대로 그리면 캔버스가 오염(tainted)되어 `toBlob()`이 막힘. 서버가 대신 fetch해서 같은 출처로 클라이언트에 넘겨주면 이 문제가 사라짐
  - 서버는 바이트만 그대로 전달(리사이즈 안 함) — R-3 결정(압축은 클라이언트에서) 유지, `sharp` 등 신규 이미지 처리 의존성 추가 없음
  - 안전장치: `http`/`https` 프로토콜만 허용, 응답 `Content-Type`이 `image/*`가 아니면 거부, 응답 크기 상한(예: 20MB) 초과 시 거부
  - 인증: 기존 `middleware.ts`의 matcher(`/((?!api/auth|login|auth/error|_next/static|_next/image|favicon.ico).*)`)가 `/api/admin/*` 전체를 이미 커버하므로 별도 인증 코드 불필요

### 클라이언트

- `src/lib/image-compress.ts` 신규 — `ImageDropzone.tsx`의 `compressImage`(File 대신 Blob 입력으로 일반화)·`blobToBase64`·`sanitizeBaseName`을 이곳으로 추출, 기존 컴포넌트와 신규 컴포넌트가 공유
- `ReadmeImageGallery.tsx` 신규 (`ProjectForm.tsx` 내 `ImageDropzone` 위에 배치, `mode` 무관하게 공통 렌더링)
  - `github` 필드 값을 `watch`해서 버튼 활성/비활성(빈 값이면 ImageDropzone의 slug 미입력 패턴과 동일하게 안내 문구)
  - 버튼 클릭 → `/api/admin/github-import/images` 호출 → 가로 스크롤 썸네일 스트립(raw URL 직접 미리보기)
  - 썸네일 클릭 → `ImageDetailModal`(자체 구현, 외부 라이브러리 없음): 확대 이미지 + alt 캡션 + 이전/다음 내비게이션 + "채택" 버튼 + 닫기
  - "채택" → `/api/admin/github-import/fetch-image` 프록시 호출 → 받은 바이트를 `image-compress.ts` 파이프라인에 통과 → `StagedImage`로 변환 → 부모의 `images` state(`ProjectForm.tsx`)에 합류 — 이후로는 `ImageDropzone`이 렌더링하는 기존 목록과 완전히 동일하게 취급(본문 삽입/경로 복사/삭제)
  - 이미 채택한 후보는 썸네일에 체크 표시(중복 채택 자체는 막지 않음, 단순화)
