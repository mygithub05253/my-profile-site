## 1. 발견 결함 및 보완 (F-1 ~ F-10)

### 심각도 상 (구현 시 오동작 유발)

**F-1. projects 상태 enum 충돌** — `validate_frontmatter.py`의 `ENUM_FIELDS["status"]`는 전 타입 공통으로 `{draft, ready, published, synced}`를 허용하는데, projects 설계(스키마 §4)는 `draft | published` 2단계다. 이대로면 프로젝트에 `status: ready`가 들어와도 L1 통과 → 사이트 노출 조건과 어긋나는 유령 상태 발생.
→ **조치**: 스키마 §4 CHECK에 **P-5(type=project ⇒ status ∈ {draft, published})** 신설, PR-A 요건에 type 조건부 enum 검증 명시.

**F-2. slug 유일성 스캔 범위 누락** — 스키마 §6은 "전 컬렉션(posts+drafts+projects) 공통 네임스페이스 UNIQUE"를 선언했지만, as-built `collect_targets()`는 `posts/`(+`--all` 시 `drafts/`)만 스캔한다. projects와 posts의 slug 충돌(예: 블로그 회고 글과 프로젝트가 같은 slug)이 감지되지 않음.
→ **조치**: PR-A 요건에 "projects/를 기본 스캔 대상에 포함 + 전 컬렉션 slug 중복 검사" 명시 (스키마 §9 L1 행 갱신).

**F-3. type:project 글의 블로그 혼입 경로** — Velite posts 컬렉션 pattern은 `posts/**/*.md`이고 type enum에 `project`가 포함되어 있어, 누군가 `posts/`에 `type: project` 파일을 두면 **블로그 목록에 프로젝트가 노출**된다.
→ **조치**: L1 CHECK **P-7(type=project ⇒ 경로는 projects/ 필수, 역으로 projects/ 내 type≠project 금지)** 신설.

### 심각도 중 (목표·정책 모순)

**F-4. "코드 수정 없이 관리" 목표 vs 정적 데이터 위치 모순** — 관리자 목표는 콘텐츠를 코드 수정 없이 관리하는 것인데, records/stacks/profile(Hero 문구·타임라인)은 사이트 레포 `src/data/`(코드)에 있어 수정에 PR이 필요하다. → **D-1 결정 필요**.

**F-5. admin 저장 브랜치 기본안의 UX 모순** — 기본안 `admin/*` 브랜치+PR은 Git 규칙과 일관되지만, **편집 1회마다 PR 수동 병합**이 필요해 "용이한 CRUD"(UC-04) 목표를 훼손한다. 반대로 main 직접 커밋은 L1 CI 게이트(PR에서만 동작)를 우회한다.
→ **재프레임(권장안)**: **PR 생성 + auto-merge 활성화** — CI(L1) 통과 시 자동 병합. Git 규칙·품질 게이트·무개입 UX 3가지를 동시 충족. → **D-2 확인 필요**.

**F-6. deploy-site 재빌드 트리거 경로에 projects 미포함 가능성** — 재빌드 트리거가 posts 변경 기준으로 배선되어 있다면 프로젝트 수정이 사이트에 반영되지 않는다. → **조치**: PR-A 요건에 "deploy-site 트리거 경로에 `projects/**` 포함, `drafts/**` 제외" 명시 (API 문서 E-3 갱신).

**F-7. velog-backup 역방향 PR의 projects 오염 방지** — 주간 역방향 diff는 velog 글(posts) 한정이어야 함을 계약으로 명시 (projects는 velog에 없으므로 diff 대상에서 구조적으로 제외되지만, 경로 명시가 안전). → API 문서 E-5 갱신.

### 심각도 하 (명시화·운영 노트)

**F-8. 이전/다음 내비 기준** — 전역 정렬(featured→order) 기준이며 목록의 현재 필터와 무관함을 "비목표"로 확정 (혼동 방지).
**F-9. sitemap 범위** — projects 11건 포함 + 한글 slug `encodeURIComponent` 일관 적용 명시.
**F-10. OG 캐시 지연** — SNS 크롤러 캐시로 썸네일 갱신이 늦게 반영될 수 있음 — 운영 노트 (장애 아님).
